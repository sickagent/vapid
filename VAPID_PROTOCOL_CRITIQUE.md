# VAPID Protocol Technical Review

## Назначение документа

Настоящий документ содержит техническую оценку исходной версии спецификации `VAPID Protocol v1.0.0`. Анализ выполнен с точки зрения межреализационной совместимости, формальной определенности и пригодности к нормативной стандартизации.

## Сводный вывод

Исходная спецификация содержит значительное количество полезных проектных идей, однако в текущем виде объединяет несколько разнородных уровней описания:

- wire format;
- модель безопасности;
- orchestration runtime;
- storage policy;
- governance;
- reputation и staking;
- zero-knowledge roadmap.

Такая структура затрудняет формирование строгого межоперабельного стандарта и увеличивает риск расхождения независимых реализаций.

## Критические замечания

### 1. Несогласованная модель идентификаторов

Одна и та же сущность адресуется различными способами:

- `Arena.arenaId` имеет формат `arena_<hash>`;
- `Message.header.arena` ожидает `DID`;
- `Artifact.arena` и `Snapshot.arena` также используют `DID`.

Следствие:

- отсутствует единый адресный слой;
- затрудняется разрешение ссылок между объектами;
- повышается вероятность несовместимой интерпретации идентификаторов.

Рекомендуемое решение:

- использовать единый тип идентификатора для `arena`;
- либо перевести `arena` на DID;
- либо использовать единый typed identifier или `URN`.

### 2. Недостаточная формализация криптографических операций

В спецификации перечислены криптографические алгоритмы, однако не определены:

- точное байтовое представление подписываемых данных;
- правила канонизации JSON;
- порядок вычисления `Hash`;
- состав данных, входящих в `stateRoot` и `merkleRoot`;
- правила кодирования подписи для каждого алгоритма.

Следствие:

- независимые реализации не смогут гарантированно проверять подписи и хэши одинаковым образом.

Рекомендуемое решение:

- зафиксировать обязательный signing suite для `v1`;
- определить каноническую сериализацию, например `JCS` для JSON или `COSE/CBOR`;
- оформить hashing and signing rules как отдельный нормативный раздел.

### 3. Ограниченная модель кодирования подписи

`Signature.value` ограничен регулярным выражением на `128` hex-символов. Такое ограничение не является универсальным для всех заявленных алгоритмов:

- `Ed25519`;
- `ECDSA-P256`;
- `BLS12-381`.

Следствие:

- схема искусственно связывает формат подписи с частным представлением;
- затрудняется расширение набора алгоритмов без изменения wire format.

Рекомендуемое решение:

- оставить один обязательный алгоритм для `v1`;
- либо сделать формат подписи алгоритм-зависимым;
- либо использовать `multibase` или `base64url` с отдельными правилами кодирования.

### 4. Несоответствие narrative-части и JSON Schema

В описательной части указано, что `Capability Tokens` требуют подпись, а `ZK` является опциональным механизмом. При этом поле `CapabilityToken.proof` отсутствует в `required`.

Следствие:

- схема допускает существование формально валидного токена без криптографического подтверждения подлинности.

Рекомендуемое решение:

- сделать подпись обязательной в схеме;
- при необходимости оставить `ZK` как дополнительное поле поверх базовой подписи.

### 5. Необязательность подписи у критичных сущностей

Для протокола, ориентированного на проверяемое взаимодействие, подпись должна быть обязательной для объектов, пересекающих границу доверия. В исходной спецификации это требование не обеспечено:

- `Arena.signature` не является обязательным;
- `Artifact.signature` не является обязательным;
- `Snapshot.signature` не является обязательным;
- `SystemPrompt.signature` не является обязательным;
- error responses допускаются вне подписанного канала.

Рекомендуемое решение:

- определить обязательный signed envelope;
- требовать подпись для всех объектов, участвующих в принятии доверительных решений.

### 6. Недостаточная формализация консенсусного слоя

В `consensusConfig` перечислены:

- `simple-majority`;
- `unanimous`;
- `pbft`;
- `raft`.

При этом `ConsensusMessage` определяет только общий контейнер и не описывает:

- модель отказов;
- требования к quorum;
- порядок выбора лидера;
- переходы между состояниями протокола;
- условия finality.

Рекомендуемое решение:

- исключить консенсус из обязательной области `Core v1`, если он не требуется для всех сценариев;
- либо выбрать один алгоритм и описать его как полноценную нормативную подсистему.

### 7. Недостаточная определенность семантики payload

`Message.header.type` перечисляет виды сообщений, однако `payload.data` остается практически неограниченным.

Следствие:

- две реализации могут принимать один и тот же документ как валидный, но интерпретировать его различным образом.

Рекомендуемое решение:

- определить отдельные схемы payload для каждого `message type`;
- либо использовать `discriminator`-ориентированную модель с жесткой типизацией.

### 8. Дублирование транспортных представлений

На верхнем уровне определены отдельные формы:

- `message`;
- `artifact`;
- `snapshot`;
- `audit`;
- `consensus`.

При этом внутри `Message` уже присутствуют типы, связанные с `artifact`, `audit` и `consensus`.

Следствие:

- формируются две параллельные транспортные модели;
- повышается вероятность появления несовместимых реализационных трактовок.

Рекомендуемое решение:

- использовать единый способ передачи;
- предпочтительно стандартизировать единый `envelope` с типизированным payload.

## Архитектурные риски

### 9. Избыточная область охвата первого релиза

Исходный документ одновременно описывает:

- identity;
- authorization;
- orchestration;
- consensus;
- storage tiering;
- payment config;
- reputation;
- staking;
- prompt governance;
- `ZK`.

Следствие:

- увеличивается объем нормативного текста;
- снижается реализуемость первого релиза;
- усложняется тестирование совместимости.

Рекомендуемое решение:

- сократить `Core v1` до минимального протокола обмена;
- вынести остальные механизмы в `extensions` или `profiles`.

### 10. Преждевременная стандартизация SystemPrompt

`SystemPrompt` зависит от:

- конкретной модели;
- внутренней оркестрации;
- политики конфиденциальности;
- реализации платформы исполнения.

Следствие:

- включение `SystemPrompt` в обязательное ядро создает избыточную связанность стандарта с runtime-предпосылками.

Рекомендуемое решение:

- вынести `SystemPrompt` в отдельное расширение;
- оставить в `Core` только ссылку на policy или execution profile.

### 11. Включение динамического состояния в переносимые объекты

Поля `revoked`, `revokedAt`, `status`, `reputation` и stake-параметры отражают изменяемое состояние.

Следствие:

- локальная копия документа не является надежным источником актуального статуса;
- возникает риск рассогласования между кэшированными копиями и реальным состоянием.

Рекомендуемое решение:

- вынести динамическое состояние в отдельные реестры или status endpoints;
- не использовать переносимый документ как единственный источник истины для отзыва и статуса.

### 12. Частичная совместимость с DID Core

`AgentIdentity` концептуально близок к DID Document, однако содержит собственные упрощения:

- используется `services` вместо стандартного `service`;
- отсутствует явное разделение `authentication`, `assertionMethod`, `keyAgreement`;
- `did:vapid` не описан как самостоятельная `DID Method Specification`.

Рекомендуемое решение:

- либо явно определить данный документ как отдельный `VAPID Identity Document`;
- либо привести структуру к требованиям `DID Core` и отдельно стандартизировать `did:vapid`.

## Замечания по JSON Schema

### 13. Корневой `oneOf` без строгой изоляции вариантов

Во многих верхнеуровневых объектах отсутствует `additionalProperties: false`.

Следствие:

- возрастают риски неоднозначной валидации;
- лишние поля могут проходить проверку без явного статуса;
- усложняется генерация типобезопасных моделей.

Рекомендуемое решение:

- использовать `additionalProperties: false` для всех закрытых объектов;
- вводить более явные discriminators или единый `envelope kind`.

### 14. Некорректные ожидания от `default`

В `JSON Schema` значение `default` обычно является аннотацией и не означает автоматическое заполнение поля валидатором.

Следствие:

- разработчики реализаций могут неверно интерпретировать наличие `default` как часть обязательной логики исполнения.

Рекомендуемое решение:

- описывать processing defaults отдельно;
- не связывать наличие `default` с автоматическим поведением валидатора.

### 15. Дублирование семантики в полях capability

В примерах `CapabilityToken.capabilities[].resource` уже содержит действия вида `read` и `write`, при том что рядом имеется отдельное поле `action`.

Следствие:

- повышается вероятность конфликтов интерпретации;
- усложняется однозначная проверка полномочий.

Рекомендуемое решение:

- хранить ресурс и действие раздельно;
- не кодировать действие внутри строки ресурса.

## Терминологическое замечание

### 16. Конфликт названия с существующим термином VAPID

Термин `VAPID` уже используется в контексте `Voluntary Application Server Identification` для `Web Push`.

Следствие:

- возникает риск путаницы в поисковой выдаче, документации и пакетах;
- усложняется позиционирование стандарта в публичном пространстве.

Рекомендуемое решение:

- либо рассмотреть переименование протокола;
- либо документировать различие между двумя значениями термина на уровне официальных материалов.

### 16.1 Рекомендация по выбору наименования

С точки зрения публикации открытого стандарта предпочтительно принять решение по наименованию до выпуска стабильной версии.

Варианты решения:

- сохранить текущее имя как рабочее и сопровождать его обязательным пояснением во всех публичных материалах;
- выполнить переименование до стадии стабильного релиза.

С точки зрения терминологической чистоты и снижения риска путаницы более предпочтительным является переименование до публикации финальной версии.

### 16.2 Критерии выбора нового имени

При выборе нового наименования рекомендуется использовать следующие критерии:

- отсутствие конфликта с существующими интернет-стандартами и общеупотребимыми протокольными акронимами;
- отражение предметной области агентного взаимодействия;
- отражение верифицируемости и целостности как базовых свойств протокола;
- пригодность для использования в репозиториях, схемах, доменных именах и пакетах;
- устойчивость наименования при дальнейшем расширении спецификации.

### 16.3 Возможные варианты переименования

В качестве технически приемлемых вариантов могут рассматриваться:

- `VAIP` = `Verified Agent Interaction Protocol`;
- `AICP` = `Agent Integrity and Coordination Protocol`;
- `DAIP` = `Decentralized Agent Interaction Protocol`.

Предварительная оценка:

- `VAIP` наиболее близок по смыслу к текущему названию и в наибольшей степени сохраняет акцент на верифицируемом агентном взаимодействии;
- `AICP` подчеркивает координацию и целостность, но в меньшей степени отражает проверяемость;
- `DAIP` подчеркивает децентрализованное развертывание, но ослабляет акцент на криптографической верифицируемости.

С точки зрения баланса семантики и пригодности к публичному использованию `VAIP` представляется наиболее нейтральным кандидатом на замену текущего имени, если решение о переименовании будет принято.

## Положительные элементы проекта

В исходной версии спецификации присутствуют значимые проектные решения:

- ориентация на capability-based security;
- внимание к provenance и auditability;
- попытка разделить агента, сообщение и артефакт;
- наличие задела для последующих расширений;
- учет operational concerns, а не только базового сценария обмена.

## Рекомендуемый состав Core v1

Для обеспечения реализуемости и формальной определенности в состав ядра рекомендуется включать только:

- `Identity`;
- `Capability`;
- `Envelope`;
- `ArtifactRef`;
- `Error`;
- `ProtocolVersion`.

Следующие компоненты рекомендуется выносить в расширения:

- consensus;
- audit;
- reputation;
- staking;
- storage tiering;
- payment config;
- zk-proofs;
- system prompt governance.

## Рекомендация по публикационному статусу

С учетом объема необходимых корректировок более обоснованными выглядят следующие варианты публикационного статуса:

- `0.3.0` как развивающаяся спецификация;
- `1.0.0-rc1` как кандидат в стабильный релиз после устранения критических замечаний;
- `Core v1.0.0` с отдельными experimental extensions.

Публикация исходного документа в статусе финального `1.0.0` представляется преждевременной до фиксации строгой базовой модели и нормативных правил обработки.
