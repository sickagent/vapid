# VAPID Security Model

## Статус

- Название: `VAPID Security Model`
- Версия: `1.0.0-draft`
- Дата: `2026-03-25`
- Статус: `Normative Companion to Core Protocol`

Документ определяет требования к обеспечению:

- подлинности отправителя;
- целостности сообщения;
- проверки полномочий;
- защиты от повторного воспроизведения сообщений;
- управляемой ротации и отзыва ключей;
- операционной интерпретации уровней доверия.

Документ является обязательным дополнением к [VAPID_CORE_PROTOCOL_STANDARD.md](./VAPID_CORE_PROTOCOL_STANDARD.md).

## 1. Цели безопасности

`VAPID Core v1` должен обеспечивать:

- `authenticity`: возможность однозначно установить подписанта сообщения;
- `integrity`: возможность подтвердить отсутствие изменений после подписи;
- `authorization`: возможность проверить наличие допустимых полномочий на выполнение действия;
- `replay resistance`: возможность выявить повторное использование ранее принятого сообщения;
- `key agility`: возможность ротировать ключевой материал без изменения идентичности агента.

`VAPID Core v1` не предназначен для обеспечения следующих свойств:

- корректность или добросовестность модели;
- достоверность семантического содержания вывода;
- prompt compliance;
- Byzantine consensus;
- конфиденциальность исполнения по умолчанию;
- доверие к runtime без отдельного attestation profile.

Указанные свойства должны определяться отдельными профилями безопасности или runtime controls.

## 2. Разделение Core Protocol и Security Model

`Core Protocol` определяет:

- структуру сообщения;
- обязательные поля;
- порядок обработки сообщения.

`Security Model` определяет:

- допустимые ключевые материалы;
- правила проверки подписи;
- правила обнаружения повторного воспроизведения;
- порядок отзыва и ротации;
- допустимые доверенные корни в рамках конкретного deployment profile.

Такое разграничение необходимо для сохранения стабильности транспортного формата и независимого развития профилей доверия.

## 3. Обязательный криптографический baseline

Для обеспечения межреализационной совместимости `Core v1` устанавливает следующий обязательный baseline:

- алгоритм цифровой подписи: `Ed25519`;
- канонизация JSON: `JCS-compatible`;
- алгоритм хэширования: `SHA-256`;
- кодирование открытых ключей: `multibase`;
- кодирование подписи: `multibase`.

Дополнительные алгоритмы могут поддерживаться в виде отдельных профилей, однако не заменяют обязательный baseline.

Реализации, не поддерживающие общий baseline, не считаются совместимыми в пределах `Core v1`.

## 4. Правила подписи

Для `envelope` и `capability token` применяются следующие правила:

- подписант `MUST` использовать активный `verification method`;
- `signing input` `MUST` формироваться из канонизированного JSON с исключением `proof.value`;
- `signing input` `MUST` включать domain separation prefix, зависящий от типа подписываемого объекта;
- `digest` `MUST` вычисляться по domain-separated `signing input` до выполнения подписи;
- `proof.created` `SHOULD` соответствовать моменту формирования подписи;
- получатель `MUST` проверять, что `proof.created` находится в допустимом временном интервале.

Модификация любого подписываемого поля после формирования подписи делает объект недействительным.

Для `Core v1` используются следующие ASCII prefixes:

- `vapid:envelope:v1\n`
- `vapid:capability:v1\n`
- `vapid:artifact-descriptor:v1\n`

## 5. Разрешение идентичности

`Security Model` не фиксирует конкретный backend реестра, однако требует единый результат проверки: получатель должен иметь возможность разрешить DID отправителя в набор допустимых `verification methods`.

Минимальные требования:

- `identity record` `MUST` быть доступен по стабильному идентификатору;
- запись `MUST` содержать сведения об активных ключах;
- запись `MUST` позволять различать активные и отозванные ключи;
- получатель `MUST` использовать именно тот `verification method`, который указан в `proof.verification_method`.

Если идентификационная запись недоступна либо указанный ключ неактивен, сообщение должно быть отклонено.

## 6. Профили доверия

VAPID допускает несколько профилей доверия. Профиль доверия не должен изменять wire format `Core Protocol`.

### 6.1 Профиль на основе DID (`DID-native Profile`)

Профиль предназначен для открытых и федеративных агентных сетей.

Характеристики:

- идентичность агента выражается через DID;
- `verification methods` публикуются в DID-совместимой записи;
- доверие определяется правилами конкретного DID method и локальной политикой;
- отзыв и ротация публикуются через слой разрешения идентичности.

Этот профиль рассматривается как базовый для стандарта VAPID.

### 6.2 Корпоративный профиль PKI (`Enterprise PKI Profile`)

Профиль предназначен для корпоративных и замкнутых deployment-сценариев.

Характеристики:

- агент сохраняет VAPID-идентификатор;
- начальное доверие может устанавливаться через корпоративный `CA` или `mTLS`;
- сертификаты `X.509` `MAY` использоваться для защиты транспортного уровня, начального связывания идентичности или публикации сервисных `endpoints`;
- подпись `VAPID envelope` остается обязательной на протокольном уровне.

Сертификатный уровень может использоваться как дополнительный механизм доверия, но не заменяет подпись протокольного сообщения.

### 6.3 Профиль статических ключей (`Static Key Profile`)

Профиль предназначен для испытательных стендов и небольших закрытых систем.

Характеристики:

- используется заранее определенный набор `agent IDs` и открытых ключей;
- обновление trust set выполняется административно;
- применимость профиля ограничена сценариями с контролируемым составом участников.

Данный профиль не рекомендуется для многоарендных и открытых сетей.

### 6.4 Уровни доверия и attestation classes

Совместимость `Core v1` не должна смешиваться с уровнем доверия к исполняющей среде.

Минимально различаются следующие уровни:

- `L0 Key-Bound`: доказано только владение ключом, которым подписан envelope;
- `L1 Registry-Resolved`: ключ разрешён через допустимый identity backend и не находится в статусе `revoked`;
- `L2 Operator-Bound`: deployment дополнительно знает, какой оператор или организация отвечает за данный agent identity;
- `L3 Runtime-Attested`: к идентичности привязано внешне проверяемое свидетельство о свойствах runtime или hardware boundary.

`Core v1` требует только уровни `L0-L1` для межреализационной совместимости.

Привилегированные операции `SHOULD` определять минимально допустимый trust level в deployment profile или локальной политике.

Заявления о `TEE`, `secure enclave`, `provider-verified runtime` или аналогичных свойствах `MUST NOT` приниматься только на основании текстовых полей identity record. Для них требуется отдельное проверяемое attestation evidence.

## 7. Роль сертификатов

Использование сертификатов в VAPID формализуется следующим образом:

- `Core Protocol` не зависит от `X.509`;
- `Security Model` определяет `X.509` как опциональный профиль доверия, а не как обязательную основу протокола;
- сертификаты `MAY` использоваться для защиты транспортного уровня, service discovery или начального bootstrap доверия;
- подпись `VAPID envelope` `MUST NOT` заменяться исключительно TLS-сертификатом.

Такое разграничение:

- сохраняет независимость стандарта от конкретной PKI;
- обеспечивает применимость в DID-ориентированных и корпоративных сценариях;
- позволяет использовать единый транспортный формат при различных механизмах доверия.

## 8. Защита от повторного воспроизведения

Защита от повторного воспроизведения является обязательной частью `Security Model`.

Получатель `MUST` проверять:

- что `message_id` ранее не использовался данным отправителем;
- что `timestamp` находится в пределах допустимого рассогласования часов;
- что `expires_at`, если оно присутствует, не истек;
- что `capability token` не истек;
- что `correlation` и `parent references` не нарушают локальную политику.

Получатель `SHOULD` хранить `replay cache` не менее:

- до момента `expires_at`, если это поле присутствует;
- `24 hours`, если `expires_at` отсутствует.

## 9. Проверка capability token

Проверка `capability token` должна выполняться как отдельный обязательный шаг.

Получатель `MUST` проверить:

- подпись `issuer`;
- соответствие `subject` фактическому `actor`;
- допустимость `resource`;
- совпадение `resource` по точному строковому значению, если deployment profile не определяет normalization rules;
- наличие требуемого действия в `actions`;
- срок действия;
- соответствие `arena_scope`, если он указан;
- глубину делегирования;
- статус отзыва, если для `issuer` доступен источник статуса.

Криптографическая корректность токена не отменяет необходимость проверки политики и области действия.

## 10. Ротация ключей

`VAPID Core v1` должен поддерживать контролируемую ротацию ключей.

Рекомендуемая модель:

- агент публикует новый `verification method` до начала его использования;
- старый ключ переводится в промежуточное состояние согласно локальной модели статусов;
- отправитель использует новый ключ для вновь формируемых сообщений;
- получатель принимает старый ключ только в пределах установленного переходного интервала;
- по завершении переходного интервала старый ключ переводится в статус `revoked`.

Профиль безопасности `SHOULD` явно определять продолжительность переходного интервала.

### 10.1 Key Recovery

Восстановление доступа не означает восстановление прежнего приватного ключа. Нормативно recovery следует трактовать как **перепривязку идентичности к новому ключу**.

Рекомендуемая модель recovery:

- recovery policy определяется заранее и привязывается к identity lifecycle вне переносимого envelope;
- при подтверждённой утрате или компрометации старый ключ переводится в `revoked`;
- новый ключ публикуется как новый `verification method`;
- событие recovery `SHOULD` оставлять проверяемую цепочку преемственности между старым и новым ключом;
- сообщения, подписанные старым ключом после момента effective revocation, `MUST` отклоняться.

Если deployment не поддерживает recovery policy, утрата ключа приводит к созданию новой идентичности и административному выводу старой из эксплуатации.

## 11. Отзыв

Отзыв ключей и `capability tokens` не должен определяться исключительно полями внутри переносимых документов.

Нормативная модель отзыва:

- отзыв публикуется во внешнем статусном источнике;
- получатель получает сведения об отзыве через слой разрешения идентичности или `token status endpoint`;
- долгоживущие токены не рекомендуется использовать без механизма актуального статуса.

Если источник статуса недоступен, решение должно приниматься локальной политикой:

- `fail-closed` для привилегированных операций;
- исключения `fail-open` допускаются только как явно установленное локальное правило.

### 11.1 Freshness статуса

Получатель `SHOULD` устанавливать максимальную допустимую "свежесть" данных identity resolution и token status.

Для чувствительных операций deployment profile `SHOULD` определять:

- максимальный возраст cached identity record;
- максимальный возраст cached revocation status;
- действия при невозможности обновить статус.

Для привилегированных операций рекомендуемой политикой является `fail-closed`, если freshness status не может быть подтверждён в пределах допустимого окна.

## 12. Локальная политика и защита транспортного уровня

VAPID не заменяет меры безопасности транспортного уровня.

Deployment `SHOULD` использовать:

- `TLS` для сетевого транспорта;
- `mTLS` при необходимости взаимной аутентификации;
- сетевую сегментацию;
- ограничение входящего и исходящего трафика;
- аудит операций верификации.

Наличие защищенного транспортного канала не отменяет обязательность проверки `protocol-level proof`.

## 13. Модель угроз

`Security Model` ориентирован на противодействие следующим угрозам:

- подмена отправителя;
- изменение сообщения при передаче;
- повторное использование ранее принятого сообщения;
- применение токена вне допустимой области действия;
- `confused deputy` вследствие некорректного делегирования;
- использование отозванного ключа;
- завышение trust claims без проверяемого attestation evidence;
- `schema confusion` при обработке непонятых расширений.

`Core v1` не предназначен для устранения следующих классов рисков:

- вредоносное содержимое, корректно подписанное доверенным агентом;
- внутренняя компрометация модели;
- side-channel leakage;
- вредоносный код внутри артефактов;
- экономические атаки на системы репутации и staking.

## 14. Обязательная последовательность проверок безопасности

Для каждого полученного `envelope` получатель `MUST` выполнять:

1. `Resolve identity`.
2. `Select verification method`.
3. `Check key status`.
4. `Canonicalize message`.
5. `Verify digest`.
6. `Verify signature`.
7. `Check timestamp and expiry`.
8. `Check replay cache`.
9. `Validate capability tokens`.
10. `Validate critical extensions`.
11. `Apply local policy`.
12. Только после этого обрабатывать `payload`.

Нарушение указанной последовательности повышает вероятность ошибок класса `parser-before-auth` и некорректной авторизации.

## 15. Рекомендации по публикации стандарта

В публичной документации целесообразно явно зафиксировать следующие положения:

- требования безопасности являются обязательной частью стандарта;
- сертификаты `X.509` не входят в обязательную область `Core Protocol`;
- DID-ориентированный профиль является базовым профилем доверия;
- корпоративный сертификатный профиль поддерживается как совместимое расширение;
- механизмы `ZK` и `TEE` рассматриваются как отдельные профили, а не как обязательный baseline.

Указанный подход обеспечивает технологическую нейтральность, формальную определенность требований и применимость стандарта в различных классах развертывания.
