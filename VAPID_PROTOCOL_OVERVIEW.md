# VAPID Protocol Document Set

## Статус

Настоящий комплект документов определяет структуру стандарта VAPID для взаимодействия ИИ-агентов. Документация разделена на отдельные нормативные и аналитические компоненты:

- `Core Protocol`: минимальный межреализационный протокол обмена;
- `Security Model`: модель доверия, управления ключами и проверки;
- `Technical Review`: технический анализ исходной версии спецификации и перечень выявленных замечаний;
- `Gap List`: перечень оставшихся архитектурных и операционных решений для публичного релиза;
- `Platform Architecture`: архитектура системы, которая имплементирует протокол.

Такое разделение обеспечивает разграничение между обязательными требованиями совместимости и сопутствующими аспектами проектирования.

Важно различать два уровня:

- `VAPID Protocol` — как агенты должны обмениваться совместимыми сообщениями;
- `VAPID Platform` — как пользователи, tenant'ы, registry, gate, worker и другие сервисы строят экосистему поверх протокола.

## Расшифровка акронима

Рабочая расшифровка наименования протокола:

`VAPID` = `Verified Agent Protocol with Integrity and Decentralization`

Семантическая интерпретация компонентов акронима:

| Символ | Компонент | Интерпретация |
|--------|-----------|---------------|
| `V` | `Verified` | Действия, сообщения и артефакты должны допускать криптографическую проверку. |
| `A` | `Agent` | Протокол предназначен для взаимодействия программных агентов, включая ИИ-агентов. |
| `P` | `Protocol` | Определяется стандартизированный формат обмена, обработки и верификации. |
| `I` | `Integrity` | Предусматриваются гарантии целостности данных, сообщений и артефактов. |
| `D` | `Decentralization` | Поддерживаются децентрализованные механизмы идентификации и федеративные сценарии развертывания. |

Союз `with` в полной формулировке является связующим элементом и не участвует в буквенном отображении акронима.

## Терминологическое замечание по наименованию

Наименование `VAPID` используется в настоящем комплекте документов как рабочее обозначение протокола. При подготовке публичного стабильного релиза рекомендуется отдельно оценить вопрос сохранения или изменения имени, поскольку термин `VAPID` уже применяется в контексте `Web Push`.

До принятия окончательного решения официальные материалы должны содержать явное указание на то, что данная спецификация не относится к `Voluntary Application Server Identification for Web Push`.

## Состав документов

### 1. Core Protocol

Файл: `VAPID_CORE_PROTOCOL_STANDARD.md`

Документ определяет:

- обязательные сущности ядра;
- структуру подписанного `envelope`;
- механизм расширений и версионирования;
- последовательность проверки сообщений;
- требования к межреализационной совместимости;
- явные границы `Core v1` по delivery semantics, retries и cancellation.

В состав `Core Protocol` включены только те элементы, которые необходимы для обеспечения единообразного обмена данными между независимыми реализациями.

### 2. Security Model

Файл: `VAPID_SECURITY_MODEL.md`

Документ определяет:

- модель доверия;
- правила подписи и канонизации;
- управление жизненным циклом ключей;
- правила отзыва;
- защиту от повторного воспроизведения сообщений;
- профили доверия, включая DID-ориентированный профиль и опциональный профиль на основе `X.509`;
- различие между совместимостью по ключу и доверием к runtime;
- правила recovery и freshness статусной информации.

`Core Protocol` не включает детализированное описание сертификатной инфраструктуры, однако содержит нормативную ссылку на `Security Model`, поскольку требования к проверке и авторизации являются обязательной частью стандарта.

### 3. Technical Review

Файл: `VAPID_PROTOCOL_CRITIQUE.md`

Документ носит аналитический характер и содержит:

- перечень выявленных несоответствий и архитектурных рисков;
- обоснование корректировок структуры стандарта;
- рекомендации по дальнейшему развитию спецификации.

### 4. Gap List

Файл: `VAPID_PROTOCOL_GAP_LIST.md`

Документ фиксирует вопросы, которые ещё не должны случайно попасть в ядро, но уже требуют явного проектного решения:

- federation trust anchors;
- payload/schema governance;
- application-level idempotency;
- attestation evidence;
- conformance suite;
- boundaries для `P2P`, `Human Escalation`, `Snapshots` и других расширений.

### 5. Platform Architecture

Файл: `VAPID_PLATFORM_ARCHITECTURE.md`

Документ описывает систему, которая реализует протокол в прикладной среде:

- регистрацию пользователей и tenant'ов;
- ownership model агентов;
- distinction между `hosted`, `connected` и `federated` агентами;
- роли `gate`, `core`, `worker`;
- внутреннее взаимодействие через `NATS`;
- хранение метаданных в `MongoDB`.

## Область Core v1

В состав `Core v1` рекомендуется включать только следующие элементы:

- идентификаторы участников и объектов;
- ссылки на идентификационные записи;
- capability tokens;
- подписанный `message envelope`;
- `artifact descriptor`;
- модель ошибок;
- правила расширения и версионирования.

## Элементы вне Core v1

Следующие подсистемы не рекомендуется включать в обязательную область `Core v1`:

- алгоритмы консенсуса;
- системы репутации;
- staking;
- платежные механизмы;
- политики tiered storage;
- prompt governance;
- zero-knowledge proofs;
- TEE attestations;
- blockchain anchoring.

Указанные возможности целесообразно оформлять как отдельные `extensions`, `profiles` или companion specifications.

## Принцип структурирования стандарта

Структура стандарта должна различать:

- требования, обязательные для всех реализаций;
- требования, зависящие от профиля развертывания;
- механизмы, сохраняющие экспериментальный статус.

В текущей редакции VAPID это достигается следующими решениями:

- ядро протокола сведено к минимально необходимому набору сущностей;
- требования безопасности вынесены в отдельный нормативный документ;
- профили доверия не включаются в описание формата обмена;
- динамическое состояние не рассматривается как часть переносимых документов, если для него требуется внешний источник актуальности;
- открытые архитектурные вопросы фиксируются в отдельном gap-документе, а не маскируются под "почти готовый" стандарт;
- архитектура системы-имплементации вынесена из протокольного ядра в отдельный companion-документ.

## Рекомендуемая последовательность внедрения

Минимальная реализация `Core v1` должна поддерживать:

1. разбор и валидацию подписанного JSON `envelope`;
2. проверку подписи в соответствии с `Security Model`;
3. проверку `capability token` и его области действия;
4. обработку `task.request`, `task.result`, `artifact.publish`, `system.error`;
5. работу с `Artifact Descriptor` как с отдельной проверяемой сущностью.

После реализации базового уровня совместимости могут быть добавлены дополнительные профили:

- профиль доверия на основе DID;
- корпоративный профиль PKI;
- расширение аудита;
- расширение консенсуса;
- расширение для ZK-механизмов.

## Рекомендуемая публикационная структура

Для публикации стандарта целесообразно использовать следующую структуру:

- `VAPID Core Protocol v1.0.0`
- `VAPID Security Model v1.0.0`
- `VAPID Gap List v0.x`
- `VAPID Platform Architecture v0.x`
- `VAPID Extensions` с независимым версионированием

Такой подход снижает связанность документации и упрощает сопровождение отдельных частей стандарта.

## Заключение

Представленная структура документации ориентирована на формализацию минимального межоперабельного ядра, отделение требований безопасности от транспортного уровня и независимое развитие дополнительных подсистем. Такой подход облегчает реализацию, проверку соответствия и последующее развитие стандарта.
